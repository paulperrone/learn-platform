## Summary

Expand learner profiles from 10 to ~20 to cover key demographic, behavioral, and scheduling gaps. Implement 5 simulation maturity levels from 5-session regression checks to 360+ session year-long simulations. Add run data cleanup tooling to prevent bloat. Validate the adaptive engine at each timescale, proving systems work from quick checks through long-term learning progressions.

**Depends on:** Plan 017.7 Phase 2 ✅ (evaluation engine)

**Motivation:** If the core engine doesn't work, being free doesn't matter — we're not effective. Simulations are our primary quality signal for the engine. Broader profiles and longer simulations surface issues that short runs with limited personas miss: review queue scaling, long-term FIRe compression, mastery plateaus, scheduling irregularity handling, and age-range coverage beyond elementary school.

**Signal separation (DECISIONS.md 2026-03-09):** Simulations validate the **engine** only. Content quality requires live data. This plan focuses exclusively on engine validation breadth and depth.

**Cross-discipline profiles deferred to Plan 018 Phase 8** — we need actual multi-subject content before cross-discipline profiles are meaningful.

## Phase 2: Simulation Runner Enhancements
**Goal:** Support rich session scheduling models that reflect real usage habits, and add run data cleanup to prevent repository bloat.

**Key context:**
- Current runner: `simulations/src/runner.ts` — supports `sessionIntervalMs` (fixed) and `timeSchedule` (variable)
- Current runs directory: `simulations/runs/` — 311+ directories accumulate without cleanup
- `.gitignore` should exclude `simulations/runs/` (ephemeral data)
- 6 profiles have scheduling metadata in their descriptions but currently run with daily intervals: `irregular-attendance`, `returning-after-gap`, `binge-learner`, `weekend-warrior`, `fading-motivation`, `completion-driven`

1. [ ] [RSH] Audit the existing `timeSchedule` implementation in `runner.ts` — understand how `sessionIntervalMs` and `timeSchedule` interact with the simulation loop, FSRS time advancement, and snapshot recording. Document the integration points where scheduling changes are needed.

2. [ ] [IMP] Add `scheduling` field to the `LearnerProfile` type in `simulations/src/types.ts`:
   ```typescript
   scheduling?: {
     type: "daily" | "irregular" | "weekday" | "gap-and-return" | "burst" | "weekend-only" | "decay" | "completion-break";
     params?: Record<string, number>;  // type-specific parameters
   };
   ```
   Update profile JSON files for the 6 scheduling-dependent profiles with their intended scheduling config.

3. [ ] [IMP] Implement scheduling presets in the simulation runner:
   - `"daily"` — 24h fixed intervals (current default, no change needed)
   - `"irregular"` — random 1-5 day gaps between sessions (seeded PRNG for reproducibility)
   - `"weekday"` — skip weekends (existing `timeSchedule.type: "weekdays"`)
   - `"gap-and-return"` — N sessions daily, then M-day gap, then N more sessions (params: `sessions_before`, `gap_days`, `sessions_after`)
   - `"burst"` — N sessions at 4-6h intervals, then M-day gap, repeating (params: `burst_size`, `burst_interval_hours`, `gap_days_min`, `gap_days_max`)
   - `"weekend-only"` — sessions only on Sat/Sun, skip Mon-Fri (2 sessions per weekend)
   - `"decay"` — interval starts at `initial_days` (default 1), increases by `decay_rate` per session (params: `initial_days`, `decay_rate`, `max_days`)
   - `"completion-break"` — daily until a mastery event occurs in the simulation, then `break_days` gap before resuming (params: `break_days_min`, `break_days_max`, `mastery_trigger_count`)
   - All presets use seeded PRNG for deterministic gaps where randomness is involved
   - CLI `--schedule <preset>` overrides profile scheduling

4. [ ] [IMP] Implement run data cleanup tooling:
   - `just simulate-clean` — delete all run directories except the latest per profile
   - `just simulate-clean --keep N` — keep the N most recent runs per profile
   - `just simulate-clean --older-than 7d` — delete runs older than N days
   - `just simulate-clean --dry-run` — show what would be deleted without deleting
   - Ensure `simulations/runs/` is in `.gitignore` (it should be ephemeral, not committed)

5. [ ] [IMP] Add data size awareness:
   - After each simulation run, print total `simulations/runs/` size
   - Warn if total exceeds 500MB
   - Add `just simulate-size` to show per-profile and total disk usage

6. [ ] [TST] Test all scheduling presets with their target profiles:
   - `irregular-attendance` with `irregular` schedule — FSRS intervals computed correctly with variable gaps
   - `returning-after-gap` with `gap-and-return` schedule — mastery retention measured correctly after 14-day gap
   - `binge-learner` with `burst` schedule — sub-day FSRS intervals handled, review queue correct after gaps
   - `weekend-warrior` with `weekend-only` schedule — mastery survives 5-day weekly gaps
   - `fading-motivation` with `decay` schedule — FSRS handles increasing intervals, mastery plateau/decline visible
   - `completion-driven` with `completion-break` schedule — mastery events trigger breaks, staircase curve visible
   - Cleanup tool correctly prunes old runs while preserving latest

7. [ ] [VAL] Run `just simulate-all 30 42` with scheduling-aware profiles, then `just evaluate`. Compare behavioral signatures — usage-habit profiles should show meaningfully different mastery curves vs their daily-scheduling equivalents. Verify `.gitignore` excludes `simulations/runs/`. Verify cleanup tool recovers significant disk space.

**Validation:** All 8 scheduling presets work deterministically. Usage-habit profiles produce distinct mastery curves that reflect their scheduling patterns. Cleanup tooling keeps repository lean. Run data size is monitored.

| L2 (30 sessions) | Core adaptive behavior — convergence, remediation, interleaving, drift | All 10 system targets |


   - **FIRe compression trend:** Does compression ratio improve as more topics are mastered? (Expected: yes, as fewer topics need explicit review)
