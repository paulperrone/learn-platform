## Summary

Transform the manual 017 → 017.5 → 017.6 remediation cycle into a systematic, repeatable "healing loop" — analogous to gradient descent on the learning engine. Simulations are the forward pass, learning-science-grounded targets are the loss function, root-cause analysis is backpropagation, and code changes are weight updates. The system must be self-documenting and extensible: when new content, subjects, question types, or system capabilities are added, the targets, profiles, and evaluation logic evolve with them.

**Depends on:** Plan 017.5 ✅ (completed 2026-03-09 — 5 PASS, 2 WARN, 0 FAIL)

**Unblocks:** All future system changes — any code change that touches adaptive systems can be validated by running `just heal` before merging.

**Known WARN items from 017.5 (calibration targets for the healing loop):**

1. **FIRe compression: WARN (neutral, not harmful)**
   - Current: -1.4% avg compression. Target: ≥20% reduction.
   - Root cause: diagnostic materializes 40+ topics simultaneously, filling the review budget regardless of FIRe compression. FIRe due-date extensions (hours per credit) can't push topics from overdue-by-days to not-yet-due fast enough.
   - Fix direction: reduce diagnostic materialization count (e.g., only materialize topics at the placement frontier ±1 grade, not all topics below placement). This shrinks the review pool below budget, letting FIRe compression take effect.
   - Source files: `packages/api/src/services/diagnostic.ts` (materializeMastery), `packages/api/src/services/srs.ts` (compressReviews, applyFIReCredit, getSessionMix)
   - Evidence: `just simulate-fire` shows with=76 vs without=74 reviews for average-older (15 sessions). Phase 3 learnings document the structural limitation.

2. **Presentation drift: WARN (8/10 correct direction)**
   - 2 profiles drift wrong: strong-young (expected UP to intermediate, actual DOWN to primary) and average-older (expected DOWN to intermediate, actual UP to advanced).
   - strong-young root cause: 1139 remediation events generate failure signals that overwhelm accuracy signal. The student IS high-ability but remediation creates noise.
   - average-older root cause: 70% mastery, 90%+ rolling accuracy generates strong upward signal. The "expected" direction (DOWN) may be miscalibrated — the system is correctly adapting to observed performance.
   - Source files: `packages/api/src/services/content.ts` (nudgeDistribution, DRIFT_RATES), `simulations/profiles/strong-young.json`, `simulations/profiles/average-older.json`
   - Fix direction: (a) weight remediation events differently in drift signal, (b) re-evaluate profile expected directions against actual ability curves.

## Phase 2: Evaluation Engine
**Goal:** Build `evaluate.ts` that reads simulation runs + `targets.json`, computes structured pass/fail/delta results, and outputs both machine-readable JSON and human-readable markdown.

**Key context:**
- Existing analysis: `simulations/src/analyze.ts` — current metrics computation (charts, content quality, regression comparison)
- Existing regression: `simulations/src/regression.ts` — fast 3-profile regression check with ±10% tolerance
- Run output structure: `simulations/runs/<profile>-<timestamp>/` — events.jsonl, session-summaries.json, state-snapshots.json, diagnostic-result.json
- Target types from Phase 1: `simulations/src/types.ts`
- Target loader from Phase 1: `simulations/src/load-targets.ts`

**Design principles:**
- Evaluation is pure computation: reads files, computes metrics, outputs report. No side effects.
- Output includes both the numeric result AND the reasoning (which profiles contributed, which sessions showed the issue)
- Each failing system includes a "suggested investigation area" mapping to specific source files and line ranges
- Evaluation handles missing data gracefully (profiles not run, short session counts, etc.)

## Phase 3: Healing Loop Orchestrator
**Goal:** Build the epoch orchestrator that runs the full simulate → evaluate → report → checkpoint cycle, with configurable iteration limits and checkpoint frequency.

**Key context:**
- Evaluation engine from Phase 2: `simulations/src/evaluate.ts`
- Existing simulation CLI: `simulations/src/cli.ts` — runs profiles with `--all --sessions N --seed S`
- Existing justfile recipes: `just simulate-all`, `just simulate-analyze`, `just simulate-regression`
- Git workflow: the project uses conventional commits, `just test` for validation

**Design principles:**
- The orchestrator does NOT make code changes — it runs simulations, evaluates, and reports. Code changes are made by Claude Code (via the `/heal` skill) or by the user.
- Each epoch produces a timestamped report in `simulations/reports/healing/`
- Checkpoints create git commits with structured messages
- The orchestrator tracks epoch history so trends are visible (is the system improving across iterations?)

1. [ ] [IMP] Create `simulations/src/heal-loop.ts` with the epoch runner:
   ```typescript
   type HealEpoch = {
     epochNumber: number;
     timestamp: string;
     simulationSeed: number;
     sessions: number;
     evaluationResult: HealingReport;
     systemsPassCount: number;
     systemsFailCount: number;
     systemsWarnCount: number;
     deltaFromPreviousEpoch: Record<string, number>;  // per-system improvement/regression
     fixesApplied: string[];  // descriptions of code changes since last epoch
     checkpointCommit?: string;  // git SHA if checkpoint was created
   };

   type HealingHistory = {
     epochs: HealEpoch[];
     startedAt: string;
     targetVersion: number;
     currentStatus: "running" | "converged" | "stalled" | "user_review_needed";
   };
   ```

   **Epoch flow:**
   1. Run `just simulate-all --sessions 30 --seed <seed>` (full simulation)
   2. Run evaluation against targets (via evaluate.ts)
   3. Compute delta from previous epoch (if exists)
   4. Detect stalls: if no system improved across 2 consecutive epochs, flag `user_review_needed`
   5. Write epoch report to `simulations/reports/healing/epoch-<N>.json`
   6. If checkpoint interval reached: create checkpoint (see step 3)
   7. Output structured summary for Claude Code consumption

## Phase 4: Claude Code `/heal` Skill
**Goal:** Create a Claude Code skill that executes one or more healing iterations — the "AI in the loop" that reads evaluation output, traces root causes in service code, makes targeted fixes, verifies with mini-sim, and checkpoints.

**Key context:**
- Skill directory: `.claude/skills/` (currently empty)
- Existing workflow skills: `/workflow-next`, `/workflow-status`, `/workflow-plan` — follow their patterns
- Evaluation output from Phase 2: `simulations/reports/evaluation.json`

- Test locations: `packages/api/src/__tests__/` — co-located test files

**Design principles:**
- The skill is a detailed prompt, not executable code — it instructs Claude Code on the healing workflow
- Each iteration has clear entry/exit criteria
- The skill includes fallback strategies: if the obvious fix doesn't work, try alternatives before escalating
- The skill knows which source files to examine for each failing system (from evaluate.ts's investigation areas)
- The skill respects the project's conventions: `just test`, service factory pattern, strict TypeScript, etc.

1. [ ] [IMP] Create `.claude/skills/heal.md` — the main healing skill prompt:

   **Structure:**
   ```markdown
   # /heal — Run Healing Iteration

   ## Overview
   [What this skill does, when to use it]

   ## Prerequisites
   [Check that simulation runs exist, targets.json exists, etc.]

   ## Workflow

   ### Step 1: Evaluate Current State
   - Run `just heal-epoch` (or `just evaluate` if recent simulation data exists)
   - Read `simulations/reports/evaluation.json`
   - Identify the highest-priority failing system

   ### Step 2: Diagnose Root Cause
   - Read the suggested investigation areas from the evaluation report
   - Read the relevant source files
   - Trace the failing metric through the code path
   - Identify the specific code that causes the metric to miss its target

   ### Step 3: Design Fix
   - Propose a targeted code change
   - Consider: does this fix risk regressing other systems?
   - If multiple approaches exist, prefer the simplest one
   - Document the fix rationale

   ### Step 4: Implement Fix
   - Make the code change
   - Run `just test` to verify unit tests pass
   - If tests fail, fix the test issue before proceeding

   ### Step 5: Verify Fix
   - Run `just heal-verify <system> --profiles <relevant> --sessions 10`
   - Check if the target metric improved
   - If improved: proceed to next failing system or checkpoint
   - If not improved: try alternative approach (max 2 alternatives)
   - If all approaches fail: document the failure, move to next system

   ### Step 6: Checkpoint (every 3 systems fixed or on user request)
   - Run `just heal-checkpoint`
   - Report summary to user: what was fixed, what remains, overall trajectory

   ## Escalation
   [When to stop and ask the user: stalled for 2+ systems, regression detected, architectural question]

   ## System-Specific Playbooks
   [Per-system diagnosis guides — where to look, what to check, common root causes]
   ```
