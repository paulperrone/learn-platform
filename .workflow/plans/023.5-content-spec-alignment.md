## Summary

Reconcile the delta between the Plan 023 content dimension specification and the actual content in learn-content. The spec defines per-item dimension fields (presentation, contentDepth, locale, flavor) as first-class content attributes, but validation only checks them if present (conditional), the generation slash command never mentions them, and the bundle generator silently applies defaults — masking the gap entirely. All 800 topics across 3 disciplines have zero dimension fields on their problems/examples. This plan fixes enforcement, updates generation templates, backfills all content, and runs a comprehensive alignment audit.

**Depends on:** Plan 023 (R2 Content Architecture) — complete
**Blocks:** Plan 025 (Consolidated System Audit) — audit needs compliant content to validate against

## Phase 1: Enforcement & Documentation Alignment ✓
**Goal:** Fix the tooling and docs so non-compliant content can't pass validation. Close the gap between what the spec requires and what the tooling enforces.

### Context for Execution

**Root cause analysis:**
- `tools/validate-content.ts` (lines 26-43): `validateDimensions()` checks fields only `if (item.flavor !== undefined ...)` — missing fields silently pass
- `packages/shared/src/types.ts` (lines 169-182): `cognitiveDemand` is optional (`?`), dimension fields not on base types at all
- `docs/content-system.md`: Says dimensions "compose multiplicatively" but doesn't explicitly say per-item fields are required
- `.claude/commands/generate-content.md`: Never mentions dimension fields in generation instructions
- `.claude/commands/content-health.md`: Doesn't check for dimension completeness
- `validate-content.ts` (line 21): `VALID_FLAVORS = ["classic", "story", "game", "visual"]` — missing "adventure", "space" from docs
- History/ELA content uses `"recall"` cognitive demand (17 occurrences) — not in documented set: procedural, conceptual, application, reasoning, error_analysis

1. [x] [IMP] Update `tools/validate-content.ts` — change dimension validation from conditional to mandatory:
   - `validateDimensions()` must ERROR (not skip) when `presentation`, `contentDepth`, `locale`, `flavor` are missing
   - Add validation that `cognitiveDemand` is present on every problem
   - Add validation that `source` is present on every problem
   - Align `VALID_FLAVORS` with docs: add `"adventure"`, `"space"` to the valid set
   - Resolve `recall` cognitive demand: add `"recall"` to `VALID_DEMANDS` (it's a legitimate Bloom's level for context-layered disciplines — factual recall is distinct from procedural fluency)
   - Add validation that problem `type` field, if present, is a valid `AssessmentType`
   - Keep validation backward-compatible: add a `--strict` flag that enforces new rules, `--lenient` for old behavior. Make `--strict` the default after Phase 2 backfill.
   - Temporarily: add `--lenient` as default until Phase 2 completes (so existing CI doesn't break)

2. [x] [IMP] Update `packages/shared/src/types.ts` — strengthen base types:
   - Make `cognitiveDemand` required (remove `?`) on `Problem` type
   - Add `recall` to `CognitiveDemand` union type
   - Add `source` as required field on `Problem` type: `source: ContentSource`
   - Add `ContentSource = "hand-authored" | "generated" | "supplementary"` type
   - Add optional dimension fields to base `Problem` and `WorkedExample` types: `presentation?: Presentation`, `contentDepth?: ContentDepth`, `locale?: string`, `flavor?: string`
   - Keep them optional on base types (source JSON), required on `BundledProblem`/`BundledExample` (R2 output)
   - Verify `AssessmentType` includes all documented types: `text-qa`, `numerical-input`, `multi-step`, `matching`, `multi-select`, `equation-builder`

3. [x] [DOC] Update `docs/content-system.md` — remove ambiguity about per-item requirements:
   - Add explicit section: "Required Fields Per Content Item" listing all mandatory fields for problems and examples
   - Clarify: dimension fields are required in source JSON (not just bundle output)
   - Document that `recall` is a valid cognitive demand for context-layered disciplines
   - Align flavor list: document all 6 valid flavors consistently
   - Add note: "The bundle generator applies defaults for missing dimensions, but source content SHOULD always include them explicitly"

4. [x] [DOC] Update `.claude/commands/generate-content.md` — require dimension fields in generation:
   - Add to problem template: `presentation`, `contentDepth`, `locale`, `flavor`, `source`, `type` fields
   - Add to example template: `presentation`, `contentDepth`, `locale`, `flavor` fields
   - Add rule: "Every problem MUST include `cognitiveDemand`, `source`, and dimension fields"
   - Add rule: "Set `presentation` to match the topic's `defaultPresentation` from graph.json"
   - Add rule: "Set `contentDepth` to match the topic's `contentDepth` from graph.json"
   - Add rule: "Set `locale` to 'en' (default) unless generating localized content"
   - Add rule: "Set `flavor` to 'classic' (default) unless generating themed content"
   - Add rule: "Set `type` to the appropriate AssessmentType (default 'text-qa')"

5. [x] [DOC] Update `.claude/commands/content-health.md` — add dimension completeness checks:
   - Add "Dimension Completeness" section to health diagnostic
   - Check: % of problems with all dimension fields present
   - Check: % of examples with all dimension fields present
   - Check: presentation field matches topic's `defaultPresentation`
   - Check: contentDepth field matches topic's `contentDepth`
   - Flag mismatches as warnings
   - Add auto-fix option: "Missing dimensions: Apply from topic metadata"

6. [x] [IMP] Update CLAUDE.md content conventions — add dimension field requirement:
   - Add to Conventions section: "All problems must include dimension fields (presentation, contentDepth, locale, flavor) and source tracking (source, cognitiveDemand)"
   - Add: "Dimension values should match the topic's defaultPresentation and contentDepth from graph.json unless intentionally creating multi-presentation or multi-depth content"

7. [x] [TST] Test enforcement changes:
   - Unit test: `validateDimensions()` in strict mode rejects problems missing `presentation`
   - Unit test: `validateDimensions()` in strict mode rejects problems missing `cognitiveDemand`
   - Unit test: `validateDimensions()` accepts `recall` as valid cognitive demand
   - Unit test: `validateDimensions()` accepts all 6 flavors
   - Verify: `just validate-content` in lenient mode still passes on current content
   - Verify: `just typecheck` passes with updated types

**Validation:** `just validate-content --strict` correctly rejects current content (missing dimensions). `just validate-content` (lenient, temporary default) still passes. `just typecheck` passes. Docs are internally consistent about required fields.

## Phase 2: Content Backfill ✓
**Goal:** Add correct dimension fields to all 800 topic files (problems + examples) across 3 disciplines. After this phase, all content passes strict validation.

### Context for Execution

**Scale:** 800 topic files × 2 (problems + examples) = 1,600 files to update. Plus 207 generated problem files.

**Dimension values to apply:**
- `presentation`: Read from topic's `defaultPresentation` in discipline's `graph.json`
- `contentDepth`: Read from topic's `contentDepth` in discipline's `graph.json`
- `locale`: `"en"` (all current content is English)
- `flavor`: `"classic"` (all current content is classic themed)
- `source`: `"hand-authored"` for `problems/`, `"generated"` for `problems-generated/`
- `type`: `"text-qa"` (all current problems are simple Q&A format)

**Cognitive demand remap:** Existing content uses non-standard `cognitiveDemand` values that must be remapped:

| Current value | Count | Target | Disciplines |
|---|---|---|---|
| `analysis` | 710 | `reasoning` | math (648), history (62) |
| `problem-solving` | 511 | `application` | math only |
| `error-analysis` | 99 | `error_analysis` | math (85), ELA (14) |
| `synthesis` | 47 | `reasoning` | math only |

History and ELA also use `"recall"` — this is now valid after Phase 1 adds it to the schema.

**Key constraint:** This is a mechanical transform, not a content rewrite. We're adding metadata fields and remapping invalid enum values, not changing questions or answers.

1. [x] [IMP] Build backfill script (`tools/backfill-dimensions.ts`):
   - Read `graph.json` for each discipline to build topic → {defaultPresentation, contentDepth} map
   - For each problem file in `problems/`: read JSON, add dimension fields to each item, write back
   - For each example file in `examples/`: read JSON, add dimension fields to each item, write back
   - For each problem file in `problems-generated/`: same, but with `source: "generated"`
   - Add `type: "text-qa"` to all problems that lack a `type` field
   - Add `source: "hand-authored"` to all problems in `problems/` that lack a `source` field
   - **Remap non-standard `cognitiveDemand` values:**
     - `"analysis"` → `"reasoning"`
     - `"problem-solving"` → `"application"`
     - `"error-analysis"` → `"error_analysis"`
     - `"synthesis"` → `"reasoning"`
   - Preserve existing VALID field values if already set (don't overwrite)
   - Preserve JSON formatting (2-space indent, trailing newline)
   - Log summary: "Updated N files, added M dimension fields, remapped K cognitive demands, skipped J already-complete files"
   - Support `--discipline <name>` flag for targeted runs
   - Support `--dry-run` flag to preview changes without writing

2. [x] [IMP] Run backfill on math (705 topics):

## Phase 3: Generation Pipeline Hardening ✓
**Goal:** Ensure all future content generation produces spec-compliant output and the bundle pipeline warns on missing fields instead of silently defaulting.

### Context for Execution

**Key files:**
- `tools/generate-bundles.ts` (lines 100-119): `applyProblemDefaults()` and `applyExampleDefaults()` silently fill in missing dimensions
- `packages/api/src/services/content-r2.ts` (lines 183-191): `createFileContentBucket()` also applies defaults silently
- `tools/validate-content.ts`: Now enforces strict by default (from Phase 2)

1. [x] [IMP] Update `tools/generate-bundles.ts` — add missing-field warnings:
   - When `applyProblemDefaults()` fills in a missing field, log a warning: `WARN: ${problemId} missing ${field}, defaulting to ${value}`
   - Add `--strict` flag: if ANY defaults are applied, exit with non-zero (fail the build)
   - Add `--lenient` flag: apply defaults silently (old behavior, for migration)
   - Make `--strict` the default
   - Summary at end: "Applied N defaults across M items (use --lenient to suppress)"

2. [x] [IMP] Update `packages/api/src/services/content-r2.ts` — add dev-time warnings:
   - In `createFileContentBucket()`, when applying defaults to items read from filesystem, log: `[content-r2] WARN: ${itemId} missing ${field}, defaulting to ${value}`
   - Only log in non-production (check `env` or add a debug flag)
   - This catches drift during local dev and simulation runs

3. [x] [IMP] Add content validation to the learn-content commit workflow:
   - Create a `.github/workflows/validate-content.yml` (or equivalent) that runs `just validate-content` on PRs to learn-content

   - Add a `just validate-content-strict` recipe to the justfile that explicitly passes `--strict`

5. [x] [TST] Test pipeline hardening:
