## Phase 1: Mastery Convergence Fix
**Goal:** Students who consistently answer correctly should gain and retain mastery. Diagnostic-materialized mastery should not be lost on first review.

1. [ ] [RSH] Investigate the exact code path causing diagnostic mastery loss. The SRS service has `shouldMasterFinal = shouldMaster || state.mastered` which should preserve mastery, but simulation shows mastery being cleared. Trace the full flow: diagnostic materialization → warmup review selection → processReview → DB update. Identify where `state.mastered` is false when it should be true.
2. [ ] [IMP] Fix diagnostic materialization: ensure `consecutiveCorrectReviews` is set to a reasonable value (e.g., 3) when diagnostic materializes mastery, so the first review doesn't fail the criterion check.
3. [ ] [IMP] Tune mastery criterion: reduce `stability >= 14` threshold. With 1-day session intervals, FSRS stability grows slowly. Options: (a) lower to `stability >= 7`, (b) add an alternative path: `consecutiveCorrectReviews >= 5` regardless of stability, (c) use a graduated mastery model where mastery strengthens over reviews rather than being a single threshold.
4. [ ] [IMP] Add mastery preservation: once a topic is mastered, it should only lose mastery on repeated failures (e.g., 2+ consecutive incorrect), not on a single re-evaluation. Prevent mastery thrashing.
5. [ ] [TST] Add unit tests for mastery transitions: mastery gain, mastery preservation on correct review, mastery preservation on single incorrect, mastery loss on repeated failures.
6. [ ] [VAL] Re-run `just simulate average-older 10` and `just simulate strong-older 10`. Verify: (a) diagnostic mastery is preserved through first review, (b) new mastery is gained by session 10 for topics with 3+ consecutive correct, (c) no regression on 85% targeting.

**Validation:** Non-struggling profiles show monotonically increasing mastery curves. Diagnostic mastery is not lost on first review.

## Phase 2: Remediation Routing Fix
**Goal:** When a student fails on a topic, the system identifies and routes to the weak prerequisite.

1. [ ] [RSH] Analyze why remediation never triggers. The current trigger is `!wasCorrect && state.currentPhase === "independent"`. Trace the session flow for misconception-fractions profile: what phase are fraction topics in when failure occurs? Is `identifyKeyPrerequisite` being called and returning null?
2. [ ] [IMP] Broaden remediation trigger: track per-topic failure patterns across the session (not just single-attempt phase check). Trigger remediation when a topic accumulates 2+ failures within a session OR when rolling accuracy on a topic drops below 50% over recent attempts.
3. [ ] [IMP] Ensure review topics can trigger remediation: currently review topics go to `independent` phase directly, but a single failure on review may not trigger remediation. Add failure tracking that spans reviews across sessions (stored in `user_topic_state`).
4. [ ] [IMP] Validate `identifyKeyPrerequisite` returns meaningful results: test with known weak prerequisite chains (e.g., fraction-addition requires whole-number addition).
5. [ ] [TST] Add unit tests for remediation: trigger on accumulated failures, correct prerequisite identification, return-to-original-topic after remediation success.
6. [ ] [VAL] Re-run `just simulate misconception-fractions 15`. Verify: (a) remediation events occur on fraction topics, (b) routes to correct whole-number prerequisite, (c) student returns to fraction topic after remediation success.

**Validation:** Misconception-fractions profile triggers ≥5 remediation events in 15 sessions, routing to correct prerequisites ≥80% of the time.

## Phase 3: FIRe Compression Fix
**Goal:** FIRe should reduce total reviews, not increase them. Practicing a parent topic should meaningfully reduce explicit review burden on child topics.

1. [ ] [RSH] Analyze why FIRe increases reviews. Hypothesis: FIRe credit keeps topics "fresh" that would otherwise lapse out of the review cycle. Compare topic lapse rates with/without FIRe.
2. [ ] [IMP] Change FIRe credit model: instead of advancing due dates (which prevents lapsing), apply FIRe credit as a stability bonus (increases stability, extending the interval between reviews without preventing natural lapsing).
3. [ ] [IMP] Only apply FIRe credit when the parent topic review was successful (rating ≥ Good). Currently checked, but verify the threshold is appropriate.
4. [ ] [IMP] Add FIRe credit cap: limit total credit a topic can receive per day to prevent over-freshening. Topics that are naturally due should still be reviewed; FIRe should only extend intervals.
5. [ ] [TST] Add unit tests for FIRe credit: stability bonus applied correctly, credit cap enforced, no credit on failed parent review.
6. [ ] [VAL] Re-run `just simulate-fire` comparing with/without FIRe. Target: >20% reduction in explicit reviews (relaxed from 30% initial target).

**Validation:** FIRe compression ratio is positive (>20%) for ≥7/10 profiles.

## Phase 4: Interleaving & Session Mix Fix
**Goal:** Sessions should introduce new topics alongside reviews, with good strand diversity.

1. [ ] [IMP] Add minimum new-topic guarantee to session mix: at least 2 new frontier topics per session (even when review queue is large). Cap review contribution to 70% of session slots.
2. [ ] [IMP] Add strand-aware interleaving: when ordering session items, ensure no two adjacent topics share the same strand. Use a simple greedy algorithm: pick the next topic from a different strand than the previous one.
3. [ ] [IMP] Reduce warmup slot count when review queue is large: warmup (mastered topic recall) should not compete with new topic introduction.
4. [ ] [TST] Add unit tests for session mix: new-topic minimum, strand adjacency, warmup/review/new proportions.
5. [ ] [VAL] Re-run `just simulate average-older 15`. Verify: same-strand adjacency <10%, new topic introduction in ≥80% of sessions, review/new ratio between 50/50 and 70/30.

**Validation:** Same-strand adjacency <10%, review/new ratio 50-70% review.

## Phase 5: Presentation Drift Stabilization
**Goal:** Presentation weights converge toward the appropriate level without oscillating.

1. [ ] [IMP] Increase drift magnitudes: raise `DRIFT_RATES` from 0.02-0.04 to 0.04-0.08 per problem. More responsive to performance signals.
2. [ ] [IMP] Add exponential moving average smoothing: instead of directly adjusting weights per-problem, maintain a running average of "direction signal" and apply drift based on the smoothed signal. This prevents single lucky/unlucky problems from reversing drift direction.
3. [ ] [IMP] Add center-level hysteresis: once the center level changes, require a stronger signal (higher threshold) to change it back. Prevents oscillation between two levels.
4. [ ] [TST] Add unit tests for drift: convergence direction for high/low ability, smoothing prevents reversal on single problem, hysteresis prevents oscillation.
5. [ ] [VAL] Re-run `just simulate struggling-older 20` and `just simulate strong-young 20`. Verify: (a) struggling-older drifts DOWN (not up), (b) both profiles show stable center level by session 15, (c) no oscillation in last 5 sessions.

**Validation:** All profiles drift in expected direction. ≥8/10 profiles stable (no center-level change) in last 5 sessions.

## Phase 6: Diagnostic Bounds Refinement
**Goal:** Improve diagnostic placement for edge cases without breaking existing good placements.

1. [ ] [IMP] Fix upward placement bias: on incorrect answers, allow `searchLow` to decrease by 1 (not just `searchHigh`). Prevents permanent lock-in from lucky correct answers.
2. [ ] [IMP] Add confidence threshold for convergence: don't stop at MIN_QUESTIONS if `searchHigh - searchLow > 1`. Require bounds to converge to within ±1 before declaring placement complete. Add a MAX_QUESTIONS safety valve (e.g., 15).
3. [ ] [IMP] Allow bidirectional presentation seeding: if diagnostic detects performance significantly above age-expected level, seed presentation UP (not just down).
4. [ ] [TST] Add unit tests for diagnostic: bounds don't lock in, struggling profile placed correctly, presentation seeded both directions.
5. [ ] [VAL] Re-run `just simulate-diagnostic`. Verify: struggling-young placed within ±1 grade, no profile takes >15 questions, presentation seeded bidirectionally.

**Validation:** 10/10 profiles placed within ±1 grade. No bounds lock-in at 8 questions.
