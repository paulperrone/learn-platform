## Phase 3: FIRe Compression Fix
**Goal:** FIRe (Fractional Implicit Repetition) should reduce total explicit reviews by ≥20%. Currently it increases reviews by ~20% (negative compression).

**Key context:**
- SRS service: `packages/api/src/services/srs.ts` — `applyFIReCredit()` at ~line 290, BFS traversal of encompassing edges, applies credit to child topics. `compressReviews()` uses greedy set-cover to select review topics that maximize implicit coverage.
- Graph service: `packages/api/src/services/graph.ts` — `getEncompassedTopics()` returns encompassing edges with weights
- Content: `content/math-foundations/graph.json` — 133 encompassing edges (enriched in Plan 013)
- Simulation: `simulations/src/adaptive-analysis.ts` — `analyzeFIReEffectiveness()` compares with/without FIRe runs. CLI: `just simulate-fire`
- Current behavior: FIRe credit advances due dates, keeping topics "fresh" that would otherwise lapse out of the review cycle. This paradoxically increases total reviews.

1. [x] [RSH] Analyze topic lifecycle with and without FIRe. Confirmed hypothesis: old FIRe credit pulled due dates closer (advancing them to now), keeping topics perpetually fresh. With 40+ topics materialized from diagnostic, all become due simultaneously. FIRe credit kept them cycling as due indefinitely, increasing total reviews by ~20%.
2. [x] [IMP] Changed FIRe credit model from due-date advancement to due-date **extension**: instead of pulling due dates closer (more reviews), push them further into the future (fewer reviews). When parent is reviewed successfully, child topic due dates are extended, reducing how often they appear in the review queue. Extension = stability × weight × (1 - retrievability), capped at 50% of stability in days.
3. [x] [IMP] Added FIRe credit cap: extension capped at 50% of child's current stability in days. Prevents over-inflation of intervals for high-weight edges.
4. [x] [IMP] Verified FIRe credit only applies on successful parent review (rating ≥ Good). The `if (rating < Rating.Good) return` check was already in place and remains correct after Phase 1 mastery changes.
5. [x] [TST] Updated and added unit tests:
   - FIRe extends child due date further out (not closer) when parent reviewed — PASS
   - Extension cap limits bonus to 50% of stability in days — PASS
   - No credit on failed parent review (rating < Good) — PASS
   - Multi-hop credit flows with diminishing weight — PASS
   - Updated integration tests (confidence-fsrs-fire) for extension model — PASS
   - All 436 tests pass with `just test`
6. [x] [VAL] Regression baseline updated, `just test` PASS (436/436 + regression PASS).
7. [x] [VAL] **Targeted simulation gate:** Ran `just simulate-fire` (seed 42, 3 profiles × 15 sessions).
   - FIRe compression ratio: 0% (neutral — neither increasing nor decreasing reviews)
   - Target ≥20% compression: NOT MET (see analysis below)
   - No regression on mastery convergence: strong-older 97.2% at 5 sessions ✓
   - No regression on remediation: functional ✓

**Gate results:**
- **FIRe no longer increases reviews** (was +20%, now 0%) — the core regression is fixed
- 20% compression target NOT MET. Root cause: diagnostic materializes 40+ topics simultaneously, all immediately due. With this many due topics, review budget always fills regardless of compression. FIRe extensions (hours per credit) can't push topics from overdue-by-days back to not-yet-due fast enough.
- **Structural fixes made:** (a) `compressReviews` now removes covered children from `remaining` set (was a bug — children stayed in pool, budget always filled), (b) `getSessionMix` uses actual compressed review count, freeing slots for new topics
- **Blocked by Phase 4:** The 20% compression target requires Phase 4's session mix changes to reduce review dominance. Once review budget is capped and new topics are introduced, FIRe compression can start reducing explicit reviews (fewer due topics = budget not always full = compression matters).
- Unit tests: 436/436 pass ✓
- Regression baseline: PASS ✓

**Validation:** Core regression fixed (FIRe neutral instead of +20% increase). 20% compression target deferred to Phase 7 validation after Phase 4 session mix changes enable it. No regressions on previously-fixed systems.

## Phase 4: Interleaving & Session Mix Fix
**Goal:** Sessions introduce new topics alongside reviews with good strand diversity. Target: same-strand adjacency <10%, review/new ratio 50-70% review, new topics introduced in ≥80% of sessions.

**Key context:**
- SRS service: `packages/api/src/services/srs.ts` — `getSessionMix()` at ~line 526, warmup selection (~line 530), stretch items (~line 551), main mix (~line 589), interleaving (~line 619)
- Graph service: `packages/api/src/services/graph.ts` — `computeFrontier()` returns new topics ready for introduction
- Content: `content/math-foundations/graph.json` — topics have `strand` field for interleaving
- Current behavior: after diagnostic materializes 40-71 topics as due, the review queue overwhelms the mix. 99% review ratio, almost no new topics. Same-strand adjacency at 14.3%.

1. [ ] [IMP] Add minimum new-topic guarantee: reserve at least 2 session slots for new frontier topics regardless of review queue size. In `getSessionMix()`, change the allocation from `mainNewCount = mainSlots - mainReviewCount` to ensure `mainNewCount >= 2` by capping `mainReviewCount` at `mainSlots - 2`.
2. [ ] [IMP] Cap review contribution: set a hard maximum on reviews per session (e.g., 70% of total slots, not just 60% of main slots). The current 60/40 split only applies to main slots — warmup adds 3 more review topics on top. Cap total reviews (warmup + main reviews) at 70% of `count`.
3. [ ] [IMP] Add strand-aware interleaving: after assembling the session item list, apply a greedy shuffle that ensures no two adjacent topics share the same strand. Algorithm: iterate through items, if current item has same strand as previous, swap with the next item that has a different strand. O(n²) is fine for 10-item sessions.
4. [ ] [IMP] Reduce warmup slots when review queue is large: if `dueTopics.length > mainSlots`, reduce warmup from 3 to 1. Warmup (mastered topic recall) should not compete with new topic introduction.
5. [ ] [TST] Add unit tests in `packages/api/src/__tests__/srs.test.ts`:
   - New-topic minimum: at least 2 new topics even when review queue is large
   - Review cap: total reviews ≤ 70% of session size
   - Strand adjacency: no two adjacent items share a strand (when possible)
   - Warmup reduction: warmup ≤ 1 when review queue > main slots
   - Run with `just test`
6. [ ] [VAL] **Targeted simulation gate:** Run `just simulate average-older 10`, `just simulate strong-older 10`, `just simulate misconception-fractions 10` (seed 42). Verify:
   - Same-strand adjacency rate <10% across all sessions

   - Review/new ratio between 50% and 70% review
   - No regression on mastery, remediation, or FIRe (Phases 1-3 fixes preserved)

## Phase 5: Presentation Drift Stabilization
**Goal:** Presentation weights converge toward the student's appropriate level without oscillating. All profiles should drift in the expected direction and stabilize by session 15.

**Key context:**
- Content service: `packages/api/src/services/content.ts` — `nudgeDistribution()` at ~line 71, `DRIFT_RATES` constants (~line 10-20), center-level calculation
- Schema: `packages/api/src/db/schema.ts` — `userSubjectPresentation` table stores per-subject presentation weights (primary, intermediate, standard, advanced)
- Current behavior: drift rates are 0.02-0.04 per problem. With ~30 problems/session and 80-90% accuracy variance, the direction signal oscillates. Only 2/10 profiles stable at session 30. `struggling-older` drifts UP instead of DOWN.
- Simulation: `simulations/src/adaptive-analysis.ts` — `analyzePresentationDrift()` tracks weight evolution per session

1. [ ] [IMP] Increase drift magnitudes: raise `DRIFT_RATES` from 0.02-0.04 to 0.05-0.10 per problem. This makes the system more responsive to performance signals, reducing the number of sessions needed to converge.
2. [ ] [IMP] Add exponential moving average (EMA) smoothing: instead of applying raw drift per-problem, maintain a running direction signal (EMA of recent drift deltas). Apply drift based on the smoothed signal. This prevents a single lucky/unlucky problem from reversing drift direction. Store EMA state in `userSubjectPresentation` (may need a migration for a `driftSignal` column).
3. [ ] [IMP] Add center-level hysteresis: once the center level changes (e.g., standard → intermediate), require a stronger signal threshold (e.g., weight > 0.4 instead of 0.3) to change it back. This prevents oscillation between adjacent levels.
4. [ ] [TST] Add unit tests in `packages/api/src/__tests__/content.test.ts`:
   - Drift converges toward expected level for high-ability student (up) and low-ability student (down)
   - EMA smoothing prevents direction reversal on single problem outcome
   - Hysteresis prevents oscillation: center level doesn't flip-flop on balanced accuracy
   - Drift rate is fast enough to shift center within 10 sessions
   - Run with `just test`
5. [ ] [VAL] If a migration was needed for `driftSignal`, run `just db-generate` and `just db-migrate`. Update the simulation runner if it needs to initialize the new column.
6. [ ] [VAL] **Targeted simulation gate:** Run `just simulate struggling-older 20` and `just simulate strong-young 20` (seed 42). Verify:

   - `strong-young` shows upward drift from primary presentation
   - Both profiles show stable center level by session 15 (no center-level change in sessions 15-20)
   - No regression on mastery, remediation, FIRe, or interleaving (Phases 1-4 fixes preserved)

## Phase 6: Diagnostic Bounds Refinement
**Goal:** Improve diagnostic placement for edge cases (struggling-young ±2 miss) without breaking existing good placements. All 10 profiles within ±1 grade.

**Key context:**
- Diagnostic service: `packages/api/src/services/diagnostic.ts` — binary search logic in `selectNextTopic()` and `getDiagnosticResult()`, bounds update at ~line 156, `MIN_QUESTIONS = 8`
- Current behavior: `searchLow` ratchets up on correct answers but never decreases. Once `searchLow = searchHigh`, both update operations are no-ops (bounds locked). All profiles finish at exactly 8 questions. Upward placement bias for 5/10 profiles. `struggling-young` placed at grade 2 vs expected 0 (±2 miss).
- Simulation: `simulations/src/diagnostic-analysis.ts` — runs all 10 profiles through diagnostic, computes placement accuracy

1. [ ] [IMP] Fix upward placement bias: on incorrect answers, allow `searchLow` to decrease (e.g., `searchLow = Math.max(0, searchLow - 1)` when answer is incorrect and current grade ≤ searchLow). This prevents permanent lock-in from a single lucky correct answer.
2. [ ] [IMP] Add convergence confidence threshold: don't stop at `MIN_QUESTIONS` if `searchHigh - searchLow > 1`. The diagnostic should continue asking questions until bounds converge to within ±1. Add `MAX_QUESTIONS = 15` as a safety valve to prevent infinite questioning.
3. [ ] [IMP] Allow bidirectional presentation seeding: the diagnostic currently only shifts presentation DOWN when performance is below age-expected level. If the diagnostic detects performance significantly ABOVE age-expected level (e.g., age 6 performing at grade 3+), seed presentation UP (primary → intermediate). Check `buildDiagnosticResult()` for the presentation seeding logic.
4. [ ] [TST] Add unit tests in `packages/api/src/__tests__/diagnostic.test.ts`:
   - Bounds don't lock in: after 3 correct + 3 incorrect, bounds still have spread
   - Struggling profile placed within ±1: simulate low-accuracy answering, verify placement ≤ expected + 1
   - Presentation seeded bidirectionally: young high-performer gets above-primary presentation
   - MAX_QUESTIONS enforced: diagnostic terminates even if bounds haven't converged
   - Run with `just test`
5. [ ] [VAL] Update regression baseline: delete `simulations/regression-baseline.json` and re-run `just test`.
6. [ ] [VAL] **Targeted simulation gate:** Run `just simulate-diagnostic` (all 10 profiles, seed 42). Verify:
   - 10/10 profiles placed within ±1 grade of expected frontier
   - `struggling-young` specifically placed at ≤ grade 1 (currently grade 2, expected 0)
   - No profile takes >15 questions
   - Presentation seeded bidirectionally (at least 1 profile seeded UP)
   - No regression on any previously-fixed system
   - If gate fails: iterate on bounds logic, re-run. After 3 attempts, document and proceed.

## Escalation: Plan 017.6

If Phase 7 determines any system still has FAIL status, Plan 017.6 is created with this structure:

```
Plan 017.6: Deep System Investigation
- Only targets systems that still FAIL after 017.5
- Each phase: (a) deep architectural analysis with code tracing, (b) 2-3 alternative fix approaches evaluated, (c) implementation of best approach, (d) targeted simulation gate
- Final phase: full re-simulation + readiness gate
- If 017.6 also fails: escalate to user for architectural discussion before proceeding
```

This prevents infinite remediation loops — two full plan iterations (017.5 + 017.6) is the maximum before requiring human architectural review.
